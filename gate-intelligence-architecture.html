Connection lifecycle
client connects→connections.add(ws)— logged with total count
ws.on('message')→handleWebSocketMessage(ws, data)→switch on type
ws.on('close')→connections.delete(ws)
ws.on('error')→pino error log — connection stays in set until close fires
on broadcast→sockets in CLOSING/CLOSED state are deleted from connections Set automatically

⚡ Wait Time Broadcast
setInterval · every 30 seconds
Recalculates all 8 gate wait times and pushes fresh data to every connected WebSocket client in real time.
setInterval(30_000) WaitTimeService broadcastWaitTimes() Redis 10s TTL stops on SIGTERM via clearInterval
⌄
Calls WaitTimeService.getAllWaitTimes(venueId) → checks Redis cache (10s TTL) or recomputes from Postgres → broadcastWaitTimes() → iterates the connections Set, sends wait_times_updated to every OPEN socket, removes any CLOSING/CLOSED sockets from the pool. If the calculation throws, the error is logged via pino and the interval waits for the next tick.
📊 Aggregation Job
0 * * * * · hourly
Refreshes both PostgreSQL materialized views so dashboard queries stay fast without blocking readers during the refresh.
node-cron REFRESH MATERIALIZED VIEW CONCURRENTLY journey_complete conversion_funnel requires unique index to use CONCURRENTLY
⌄
Executes REFRESH MATERIALIZED VIEW CONCURRENTLY journey_complete then REFRESH MATERIALIZED VIEW CONCURRENTLY conversion_funnel. The CONCURRENTLY keyword means both views remain readable during the refresh — zero downtime for /ops/dashboard and /ops/funnel. Requires the unique indexes idx_jc_nudge and idx_cf_date to already exist — they are created by the migration scripts.
🎯 Forecast Calibration Job
0 */6 * * * · every 6 hours
Computes per-gate MAPE by comparing predicted vs actual wait times. Warns when accuracy drops below the 20% threshold.
node-cron MAPE = MAE / predicted × 100 confirmations JOIN gate_entries 6-hour window ⚠ alert if MAPE > 20% Gemini re-training hook
⌄
Joins confirmations → gate_entries on entry_token, filters to the last 6 hours, computes AVG(|predicted - actual| / predicted) × 100 per gate. If any gate's MAPE exceeds 20%, a warning is logged with the gate_id and MAPE value. This is the designed hook point for Gemini re-calibration or manual parameter tuning — the current implementation logs only. Requires real gate_entries rows (from ticket scanners writing actual wait times back to the DB).

4xx Client Errors
400
Bad Request — Missing or invalid parameters
Query or body parameters fail validation. Returned by all endpoints when required fields are absent or malformed (e.g. non-numeric lat/lng on /fans/nudge, missing user_id).
{ "error": "BAD_REQUEST", "code": 400, "message": "user_id is required" }
401
Unauthorized — Missing or invalid JWT
Authorization header is absent, JWT is malformed, signature verification fails, or the token is expired. Applies to all routes protected by validateAuth middleware: /fans, /ops, /gates.
{ "error": "UNAUTHORIZED", "code": 401, "message": "Invalid or expired token" }
404
Not Found — Resource does not exist
Returned by GET /api/gates/:id when the gate_id is not present in the database, or by any undefined route path.
{ "error": "NOT_FOUND", "code": 404, "message": "Gate gate_99 not found" }
409
Conflict — Duplicate submission
Returned by POST /api/fans/feedback when an entry_token has already been used to submit feedback (unique constraint on feedback.entry_token).
{ "error": "CONFLICT", "code": 409, "message": "Feedback already submitted for this entry token" }
410
Gone — Entry token expired
Returned by POST /api/fans/confirm when the nudge token has passed its 20-minute expiry window (confirmations.expires_at < NOW()).
{ "error": "TOKEN_EXPIRED", "code": 410, "message": "Entry token has expired" }
429
Too Many Requests — Rate limit exceeded
Client IP exceeded 100 requests in a 15-minute window. Managed by express-rate-limit. Response includes a Retry-After header.
{ "error": "RATE_LIMITED", "code": 429, "message": "Too many requests, please slow down" }
5xx Server Errors
500
Internal Server Error — Unhandled exception
Catch-all for unhandled errors in route handlers or service layer. Full stack trace is logged via pino. In production (NODE_ENV=production) the response never exposes the stack.
{ "error": "INTERNAL_ERROR", "code": 500, "message": "An unexpected error occurred" }
503
Service Unavailable — Database unreachable
Returned exclusively by GET /api/health when PostgreSQL cannot be reached. All other endpoints fall back to the in-memory mock layer and continue returning 200s.
{ "error": "DB_UNREACHABLE", "code": 503, "message": "Database connection failed", "status": "degraded" }

Client sends a WebSocket message with an unrecognised type field. The connection is kept alive; an error event is sent back to the specific client only.

{ "type": "error", "message": "Failed to process message" }
